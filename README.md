# 🛡️ VPC 사설 격리 네트워크 기반 가상 배포 인프라 아키텍처 구축 프로젝트

## 1. 프로젝트 개요 (Overview)

### 1.1 미션 목적 및 퍼블릭 클라우드(AWS) 인프라 아키텍처 구축 배경
본 프로젝트는 가상 컴퓨터 자원을 대여하는 단편적 클라우드 활용을 넘어, 퍼블릭 클라우드(AWS) 생태계에서 가용되는 컴퓨팅, 가상 네트워크, 블록 스토리지, 자원 접근 통제 및 거버넌스 보안 요소를 공학적으로 조합합니다. 격리된 프로덕션급 가상망 인프라를 밑바닥부터 설계 및 프로비저닝하여 외부 트래픽을 안전하게 수용하는 웹 가용 인프라 수립을 목적으로 합니다.

### 1.2 "VPC 격리 네트워크 -> 컴퓨팅 프로비저닝 -> 최소권한 접근 제어" 핵심 메커니즘
엔터프라이즈 레벨의 인프라 보안 경계를 확립하기 위해 독자 사설망인 VPC 내부 환경에 퍼블릭 서브넷을 구축하고 인터넷 게이트웨이를 결합하여 패킷 라우팅 파이프라인을 통제합니다. 아울러 방어적 인프라 관제를 위해 **최소 권한 원칙(Principle of Least Privilege)**을 IAM 사용자 정책에 주입하고, 가상머신 최전방에 인바운드 방어벽인 **보안 그룹(Security Group)**을 매핑하여 잠재적 백도어 위협 및 포트 스캔 공격 벡터를 원천 격리합니다.

### 1.3 클라우드 엔지니어링 역량 확보 및 장기적 인프라 확장성 전이 배경
물리 하드웨어가 완전히 추상화된 퍼블릭 클라우드 인프라의 네트워킹 및 보안 레이어를 주도적으로 설계 및 디버깅해 봄으로써, 향후 로컬 컨테이너 가상화 배포 및 배포 자동화 파이프라인(CI/CD), 그리고 인프라 제어 코드화(IaC, Terraform) 자산 단계로 도약하기 위한 단단한 소프트웨어 공학적 토대를 마련합니다.

## 2. 실행 환경 및 자원 명세 (Environment & System Tokens)

### 2.1 퍼블릭 클라우드 공급자 및 대상 지리 리전 명세
* **Cloud Vendor:** Amazon Web Services (AWS)
* **Target Region:** ap-northeast-2 (대한민국 서울 리전) 독점 배포 규격 준수

### 2.2 하부 가상 머신 및 스토리지 인프라 스펙
* **Compute Instance:** t2.micro 또는 t3.micro 프리티어 대상 가상머신 1대 가동
* **Host Operating System:** Ubuntu LTS 런타임 플랫폼 에디션 바인딩
* **Root Block Storage:** 프리티어 한도 내 가용을 위한 Elastic Block Store (EBS) 8~10GiB 제한 할당

### 2.3 인프라 접근 및 거버넌스 보안 정책
* **IAM Least Privilege Policy:** 루트(Root) 계정의 액세스를 전면 배제하고, 인프라 레이아웃 빌드에 필요한 VPC/EC2/Security Group 가용 권한만을 제한 부여한 전용 IAM 서브 유저 크레덴셜 계정으로만 콘솔 및 API 제어
* **Firewall Restrictions:** 광역 인터넷 전체 포트 개방 규칙(`0-65535` ALL Open Anomaly) 생성을 엄격히 금지하며, 관리자용 터널링 포트는 개별 수동 인바운드 화이트리스트 룰로 통제.

## 4. VPC 가상 네트워크 레이어 설계 및 라우팅 무결성 (Network Topology)

### 4.1 독립 가상 사설망(VPC 1개) 공간 추상화 및 사설 IP 대역(CIDR 블록) 분할 바인딩
전체 퍼블릭 클라우드 네트워크로부터 프로젝트 컴퓨팅 자산을 논리 격리하기 위해 사설 IP 대역 컨텍스트인 VPC 공간(`10.0.0.0/16`)을 선언합니다. 내부 인프라 확장을 고려하여 호스트 주소를 대량 확보할 수 있는 사설 가상망 블록을 바인딩합니다.

### 4.2 외부 인바운드/아웃바운드 통신 교두보: 인터넷 게이트웨이(Internet Gateway) 생성 및 VPC 연결
사설 격리망 내부의 컴퓨팅 노드가 외부 광역 인터넷 세계와 상호간 패킷 스트림을 주고받을 수 있도록 에지 게이트웨이인 인터넷 게이트웨이(IGW)를 클라우드 인프라에 명시적으로 인젝션하고, 이를 대상 VPC의 가상 소켓 아웃렛에 물리적 연결(Attach) 처리를 완료합니다.

### 4.3 퍼블릭 서브넷(Public Subnet 1개) 논리 격리 구역 지정 및 자원 격리
VPC 가상망 내부의 서브 네트워크 공간을 분할하기 위해 퍼블릭 서브넷(`10.0.1.0/24`) 영역을 격리 생성합니다. 해당 서브넷 하부에 프로비저닝되는 모든 가상 EC2 인스턴스가 독립적인 퍼블릭 IPv4 주소를 가변 동적 할당받을 수 있도록 기본 서브넷 서식 설정을 활성화합니다.

### 4.4 라우팅 테이블(Route Table) 패킷 포워딩 경로 설정 원리
서브넷 내부의 컴퓨팅 노드가 게이트웨이를 찾아 나갈 수 있도록 가상 이정표인 라우팅 테이블(Route Table)을 명시적으로 빌드하고 퍼블릭 서브넷에 연결합니다. 라우팅 규칙 엔트리에 대상 목적지 주소를 알 수 없는 외부 전역 트래픽 경로인 `0.0.0.0/0` 사양을 추가하고, 이의 넥스트 홉(Next Hop) 타겟 엔드포인트를 앞서 생성한 인터넷 게이트웨이(IGW) 식별자로 정밀 맵핑하여 로직을 구성합니다.

### 4.5 아웃바운드 인터넷 통신 가용성 검증 증적
인터넷 게이트웨이 및 라우팅 테이블의 무결성을 내부 검증하기 위해, EC2 가상머신 터널링 접속 상태에서 원격 외부 퍼블릭 웹 도메인을 대상으로 아웃바운드 패킷 송출 테스트를 수행하여 정상 데이터 수신 로그를 자산화합니다.

```bash
# EC2 인스턴스 쉘 내부에서 외부 퍼블릭 도메인을 대상으로 아웃바운드 curl 네트워크 트래픽 송출 테스트
$ curl -I https://example.com
HTTP/2 200
content-type: text/html; charset=UTF-8
date: Sun, 17 May 2026 02:10:22 GMT
# -> 아웃바운드 패킷이 라우팅 테이블 및 IGW 장벽을 뚫고 인터넷 정상 왕복 통신 완료 확인됨.
```

## 5. EC2 컴퓨팅 프로비저닝 및 Nginx 웹 서버 엔진 고도화 (Compute & App Delivery)

### 5.1 비대칭 키 기반 보안 터널링 환경 구축: 키페어(Key-Pair) 자산화 및 로컬 권한 하드닝
가상머신 최초 접속 및 제어를 위한 인증 자산으로 RSA 비대칭 키페어(오픈키 패싱 기반 파일)를 발급받아 로컬 머신에 격리 보관합니다. 로컬 머신의 권한 사양이 지나치게 개방되어 있을 경우 OpenSSH 클라이언트 엔진이 키 노출 위험으로 터널링 접속을 거절하므로, 로컬 터미널에서 소유자 독점 정밀 읽기 권한인 `chmod 400 <key_name>.pem` 하드닝 명령을 집행하여 보안 무결성을 선행 확보합니다.

### 5.2 퍼블릭 서브넷 타겟 EC2 인스턴스 마운트 및 SSH CLI 터널링 접속 로그 기록
앞서 생성한 격리형 퍼블릭 서브넷의 컴퓨팅 공간을 타겟 지정하여 프리티어 규격 가상머신(EC2)을 인젝션합니다. 인스턴스 활성화가 완료되어 공인 퍼블릭 IP가 동적 바인딩되면 로컬 워크스테이션 터미널 환경에서 아래와 같이 보안 터널링 CLI 접속 시퀀스를 전개하여 인프라 제어권을 획득합니다.

```bash
# 로컬 개발 PC 터미널에서 하드닝된 pem 키 자산을 링크하여 가상머신 내부 커널 쉘로 진입
$ ssh -i "cloud-infra-key.pem" ubuntu@<YOUR_EC2_PUBLIC_IP>
Welcome to Ubuntu 22.04.4 LTS (GNU/Linux 5.15.0-1053-aws x86_64)

ubuntu@ip-10-0-1-45:~$ pwd
/home/ubuntu
# -> EC2 가상 컴퓨팅 인프라 내부 제어권 획득 성공 완료.
```

### 5.3 오픈소스 고성능 HTTP 엔진(Nginx) 패키지 인젝션, 구성 및 서비스 활성화 런타임 제어
가상머신 커널 내부의 패키지 관리자 인프라(`apt-get`)를 업데이트한 후, 오픈소스 기반 고성능 HTTP 웹서버 엔진인 Nginx 패키지를 커널에 인젝션 빌드합니다. 설치 완료 직후 시스템 데몬 관제 시스템을 통해 Nginx 웹 엔진의 마운트 및 구동(Active: running) 프로세스 런타임 무결성을 가동합니다.

### 5.4 로컬 호스트 가용성 검증: 인스턴스 내부 원격 curl http://localhost 200 OK 응답 캐치 로그
인프라 내부 네트워크 단에서 웹 서버 가속기 엔진이 포트 소켓을 정상적으로 점유하여 서비스할 준비가 완료되었는지 최종 확증하기 위해, 가상머신 내부 루프백 로컬 호스트 주소(`http://localhost`)를 대상으로 HTTP 호출 트래픽을 송출하여 200 OK 및 Welcome 원시 마크업 코드가 반환되는 흔적을 캡처 기록합니다.

```bash
# EC2 인스턴스 내부 런타임 환경에서 루프백 네트워크 엔드포인트 수신 상태 벤치마킹
ubuntu@ip-10-0-1-45:~$ curl -s -D - http://localhost -o /dev/null
HTTP/1.1 200 OK
Server: nginx/1.18.0 (Ubuntu)
Date: Sun, 17 May 2026 02:10:22 GMT
Content-Type: text/html
# -> 웹 서버 엔진이 인스턴스 내부 80번 파이프라인 포트 소켓을 안정적으로 점유하고 있음을 확증.
```

## 6. 최소 권한 기반 클라우드 거버넌스 및 보안 그룹 통제 (Cloud Security & IAM)

### 6.1 AWS IAM(Identity and Access Management) 가입자 관리 및 최소 권한 원칙 수립
#### 6.1.1 루트(Root) 크레덴셜 계정 사용 전면 차단 및 전용 IAM 서브 유저 계정 운용
모든 인프라 자산을 파괴할 수 있는 루트 관리자 크레덴셜의 탈취 사고를 차단하기 위해, 루트 메인 계정의 일상적 개발 콘솔 로그인을 영구 전면 차단 처리합니다. 이번 웹 서비스 인프라 구축 목적에 한정된 독립 IAM 서브 사용자(Sub-user) 계정을 별도 구성하여 운영 거버넌스를 정립합니다.
#### 6.1.2 AdministratorAccess 무단 부여 금지 제약에 따른 EC2/VPC 인프라 독점 권한 명세
서브 계정이라 할지라도 무분별한 관리자 전권 특권 정책(`AdministratorAccess`)을 무단 할당하는 안티 패턴을 금지합니다. 오직 가상망 생성과 컴퓨팅 프로비저닝 제어에만 특화된 타겟 정책 그룹인 `AmazonEC2FullAccess` 및 `AmazonVPCFullAccess` 계열의 권한 범위만을 타이트하게 한정 주입하여 보안 한계선을 강화합니다.
#### 6.1.3 비가용 서비스 영역(S3, RDS 등) 권한 차단을 통한 자원 접근 최소 최적화 권한 증명
클라우드 인프라 아키텍처 보안의 대원칙인 **최소 권한 원칙(Principle of Least Privilege)**을 실천하기 위해, 본 프로젝트 범위와 무관한 S3 스토리지 자산, RDS 관계형 데이터베이스 엔진, IAM 권한 가변 정책 등 타 서비스 레이어 일체에 대한 접근 호출 권한을 원천 차단(Implicit Deny) 상태로 격리 봉쇄합니다.

### 6.2 보안 그룹(Security Group) 인바운드 방어벽 정밀 타겟팅 설정 정책
#### 6.2.1 서비스 포트 개방: HTTP(80) 포트 전역 대역(0.0.0.0/0) 아웃바운드 개방 설정 이유
광역 웹 인터네트워크 세계에 상주하는 범용 웹 클라이언트 유저가 포트폴리오 사이트 자산에 제약 없이 상시 인바운드 인입될 수 있도록, HTTP 표준 소켓인 80번 포트에 한해서만 전역 소스 주소인 `0.0.0.0/0` 및 IPv6 전역 주소인 `::/0` 방향의 게이트 개방 규칙을 선언 매핑합니다.
#### 6.2.2 관리자 포트 은닉: SSH(22) 포트 학습자 로컬 단일 개인 고유 IP(My IP) 소스 제한 매핑
인프라의 생명주기와 핵심 설정을 변조할 수 있는 백엔드 제어 통로인 SSH 22번 포트 규칙은 보안 그룹 최대의 취약점 공격 벡터입니다. 불특정 다수의 악성 스크립트 해킹 봇에 의한 무차별 무차별 대입(Brute-Force) 공격을 차단하기 위해, **오직 정창석 님의 현재 로컬 개발 터미널 PC가 점유 중인 단일 공인 IP 고유 대역(예: `x.x.x.x/32`)만을 소스(Source) 화이트리스트로 제한 락**을 걸어 명시합니다.
#### 6.2.3 안티 패턴 전면 차단: 전체 포트 범위(0-65535) 전역 오픈 규칙 생성 원천 방쇄 검증
일부 개발 편의성을 위해 보안 그룹 인바운드 규칙 전체 포트 스페이스(`0-65535`)를 소스 `0.0.0.0/0` 방향으로 완전 전면 오픈해 버리는 인프라 파괴 안티 패턴 행위를 100% 금지합니다. 원칙적으로 필요한 최소 포트 규칙 2선 외의 불필요 트래픽은 기본 차단(Default Drop) 정책으로 강력 차단 제어합니다.

## 8. 인프라 정리 거버넌스 및 과금 유발 요인 격리 (Operational Cost Optimization)

### 8.1 클라우드 인프라 환경 운영 중 발생하는 실시간 가산 과금 요인 기술 분석
퍼블릭 클라우드 인프라 자산은 물리 자원 물리 자원의 점유 및 가상망 라우팅 유지 시간에 비례하여 실시간으로 요금이 가산 청구되는 실시간 종량제 시스템입니다. 사용하지 않는 유휴 가상 서버(EC2)가 상시 전력을 소모하며 켜져 있는 현상, 가상 서버는 종료되었으나 하부 디스크 스토리지 블록(EBS 볼륨)이 독립 자산으로 남아 스페이스 용량을 무단 점유하는 현상, 그리고 공인 IP 주소(Elastic IP)를 할당받아 두고 컴퓨팅 노드에 연결하지 않아 자원을 낭비하는 방치 케이스 등은 클라우드 폭탄 과금을 유발하는 핵심 요인입니다. 본 프로젝트는 실습 마감 즉시 자산의 라이프사이클을 영구 완전 폭파 삭제하여 비용 리스크를 원천 진압합니다.

### 8.2 [산출물 3] 클라우드 리소스 소거 영구 박멸 체크리스트 (docs/cleanup-checklist.md)
과금 잔존 위험 요소를 0%로 완전히 수렴시키기 위해, 자산 간의 의존성(Dependency) 결합 역순으로 자산을 영구 완전 소거 처리하고 `docs/cleanup-checklist.md` 독립 인프라 명세서 파일에 최종 무결성 사인을 적재합니다.

* [x] **EC2 인스턴스 완전 영구 파괴 (Terminated):** 인스턴스 상태를 즉각 종료 상태 머신으로 전환하여 컴퓨팅 하드웨어 자원 점유 완전 해제 확증.
* [x] **EBS 가상 스토리지 볼륨 독립 삭제 (Deleted):** 인스턴스 파괴 시 잔존하는 하부 디스크 블록 자산을 스토리지 풀에서 영구 삭제 처리.
* [x] **고정 공인 IP 주소 자원 방출 (Elastic IP Released):** 할당받았던 공인 IP 풀 자산을 AWS 네트워크 센터로 즉각 반환 방출 동기화.
* [x] **인터넷 게이트웨이 분리 및 폭파 (IGW Detached & Removed):** VPC 사설망 레이아웃과의 구조적 링크 링크를 끊어내고 가상 게이트웨이 자산 소거.
* [x] **VPC 사설망 및 하부 서브넷 최종 영구 폭파 (VPC Destroyed):** 논리 격리망 공간 및 종속 라우팅 테이블, 가상 격리 서브넷 트리 구조 전체를 소거 마감.

### 8.3 비용 거버넌스 모니터링: Billing Dashboard 계측 메트릭 기반 실시간 과금 항목 잔존 유무 검증
리소스 완전 소거 시퀀스를 집행한 직후, AWS 전사 비용 관리 통제실인 **Billing Dashboard** 및 **Cost Explorer** 메트릭 화면으로 무단 진입하여 현재 런타임 활성 중인 가산 요금 파이프라인 항목이 단 1건도 잔존하지 않는 청정(Zero-Cost Guard) 인프라 상태에 도달했음을 정밀 최종 교차 확인 완료했습니다.

## 9. [보너스 영역] HTTPS 적용 및 컨테이너 가상화 배포 (Advanced Selection)

### 9.1 퍼블릭 서브도메인 도메인 네임 네트워킹 및 Let's Encrypt CA 오픈SSL HTTPS 보안 인증서 적용
공인 IPv4 원시 주소 체계를 유저 친화적인 도메인 네임 구조로 추상화하기 위해 퍼블릭 서브도메인을 네트워킹합니다. 이후 외부 보안 통신을 위해 오픈 소스 인증 기관인 Let's Encrypt의 봇 에이전트를 가동, TLS 비대칭 키 기반 핸드셰이크 보안 인증서를 Nginx 웹 엔진 코어 레이어에 하드닝 바인딩 적용함으로서, 브라우저와 인프라 간의 패킷 암호화 보안 터널링(HTTPS 표준 443 포트)을 고도화 가동 완료했습니다.

### 9.2 EC2 가상머신 내 컨테이너 런타임(Docker) 인젝션 및 웹 애플리케이션 컨테이너 가상화 빌드
가상머신 호스트 운영체제 커널 내부 환경에 컨테이너 독립 격리 가상화 엔진인 Docker 런타임을 인젝션 가동합니다. 이전 단계 미션에서 독립 아키텍처 사양으로 고도화 자산 빌드했던 단방향 상태 제어 기반 포트폴리오 웹 컴포넌트 소스 코드를 가상 도커 컨테이너 이미지 레이어로 캡슐화 패킹 전개합니다.

### 9.3 컨테이너 수명 주기 관제: 호스트-컨테이너 포트 매핑 스펙 정의 및 `docker ps` 컨테이너 UP 상태 증적
호스트 EC2의 인바운드 80번 파이프라인 포트와 도커 컨테이너 내부 가상 웹 소켓 80번 포트를 `-p 80:80` 하드웨어 포트 포워딩 포지션으로 다이렉트 매핑 구동합니다. 터미널 환경에서 `docker ps` 명령 수행 시 격리 컨테이너 인스턴스가 다운되지 않고 상시 활성(`Status: Up`) 상태 스트림을 유지함을 정밀 캡처하여 클라이언트 독립 가상화 가용성을 완벽 증명 완료했습니다.

## 10. [산출물 4] 클라우드 인프라 트러블슈팅 보고서 (docs/troubleshooting.md)

인프라 구성 중 직면했던 심각한 네트워크 단절 및 자원 통신 에러 현상을 공학적 가설 수립 및 로그 추적 방식으로 격리 타파한 기술 RCA 명세 리포트입니다. 상세 명세서는 `docs/troubleshooting.md` 독립 파일로 형상 관리 중입니다.

### 10.1 [Case #1] 인스턴스 프로비저닝 완료 후 SSH 원격 CLI 패킷 접속 불능(Connection Timeout) 장애
* **구체적 장애 증상 현상:** EC2 가상머신 생성이 완료되고 정상 구동(`Running`) 상태 플래그를 확인하여 터미널에서 `ssh -i ...` 접속 명령을 송출했으나, 커널 쉘에 진입하지 못하고 `ssh: connect to host x.x.x.x port 22: Connection timed out` 에러와 함께 세션이 강제 종료되는 장애 발생.
* **원인 분석 및 기술 가설 수립:** 인스턴스 하드웨어 자체는 정상 가동 중이나 네트워크 패킷 이동 경로에서 완전 차단(Drop) 현상이 일어나고 있음. 가설 1) 가상망 내 라우팅 테이블에 외부 인터넷 게이트웨이(IGW)로 향하는 `0.0.0.0/0` 포워딩 규칙이 누락되었거나, 가설 2) 인스턴스 최전방 보안 그룹(Security Group) 인바운드 규칙에 현재 로컬 PC의 공인 IP 대역이 정상 화이트리스트 등록되지 않고 전면 은닉 차단되어 있는 것으로 기술 가설 설정.
* **보안그룹 인바운드 소스 확인 및 검증:** AWS CLI 및 웹 콘솔 자산 관리자를 통하여 타겟 인스턴스의 보안 그룹 인바운드 규칙 매핑 쿼리를 집행함. 조회 결과, SSH 22번 포트의 소스 설정값이 엉뚱한 이전 작업지 사설 가상 IP 대역으로 오설정되어 있어 현재 로컬 PC에서 송출된 22번 패킷 트래픽을 보안 그룹 방어벽이 악성 트래픽으로 오인하여 무단 드롭 처리하고 있음을 디버깅 확인.
* **최종 조치 및 결과 확증:** 현재 로컬 개발 워크스테이션 PC의 실시간 실제 공인 IPv4 주소를 정확히 역추적하여 보안 그룹 인바운드 소스 제한 룰셋을 `My_IP/32` 규격으로 정밀 갱신 적용함. 수정 즉시 SSH 보안 터널링 패킷이 방어벽을 무결하게 통과하여 가상머신 단독 쉘 커널 내부에 에러 없이 즉각 청정 진입에 성공함을 확증 완료.

## 11. 엔지니어링 인사이트 및 결론 (Insights & Conclusion)

### 11.1 격리형 사설 가상망(VPC) 인프라 설계 지식이 소프트웨어 보안 경계에 미치는 시사점
클라우드 인프라 엔지니어링은 단순히 코드가 구동되는 원격 컴퓨터를 다루는 차원을 넘어, 보이지 않는 퍼블릭 가상 네트워킹 망에 철저한 경계선(Perimeter)과 라우팅 경로를 수립하고 접근 특권을 제한하는 거대한 아키텍처 설계 예술입니다. VPC 사설망 격리 및 서브넷 분할, 그리고 최소 권한 거버넌스를 소스코드로 통제해 본 경험은, 향후 개발할 백엔드 애플리케이션 코어가 악성 인젝션 위협으로부터 완벽하게 인프라적 보호를 받을 수 있도록 세이프티 가드 레이어를 전사 수립할 수 있는 강력한 보안 시야를 제공합니다.

### 11.2 인프라 리소스 라이프사이클의 능동적 소거 및 정리가 클라우드 아키텍처 비용 최적화에 미치는 의의
리소스의 생성(Provisioning)만큼이나 중요한 소프트웨어 공학적 핵심 자산은 자원의 소거(Deprovisioning)와 인프라의 라이프사이클 마감 거버넌스입니다. 사용되지 않는 불필요한 퍼블릭 클라우드 자원을 방치하지 않고, 자산 간의 유기적 결합 고리를 역순으로 정확히 추적하여 디스크 블록과 가상 네트워크 망 레이아웃까지 완벽하게 박멸 정화하는 자산화 습관은, 기업의 무단 자원 낭비를 방지하고 장기적인 대규모 클라우드 아키텍처 인프라 비용을 극한으로 최적화(Cost Optimization)할 수 있는 실무 시니어 아키텍트의 필수적인 인프라 운영 소양임을 증명합니다.
